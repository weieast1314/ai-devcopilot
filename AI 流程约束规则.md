# AI 流程约束规则

> 适用于当前项目中的 AI 编辑器协作场景，用于约束 AI 严格按照 `AI DevCopilot` 标准流程推进，并提高过程汇报密度。

## 1. 目标

本规则用于确保 AI 在处理需求、修复问题、实现代码和交付时：

- 先分析与计划，再进入实现
- 只按当前确认过的计划执行
- 发现偏差先更新计划，不擅自扩大修改范围
- 完成一项汇报一项，保持过程透明
- 在关键阶段暂停，等待用户明确确认

## 2. 适用范围

适用于以下场景：

- 标准开发流程：`/dev-flow`
- 计划生成：`进入计划模式`
- 计划执行：`确认计划，开始执行` / `执行计划`
- 热修复流程：`/hotfix`
- 飞书文档需求：`feishu-doc-to-branch`
- 文字需求：`text-to-branch`

## 3. 核心强约束

### 3.0 入口路由优先

AI 在处理用户输入时，必须优先调用 `entry-router` 进行路由判断。

要求：

1. 用户输入首先经过 `entry-router` 匹配 Pipeline trigger。
2. 禁止直接调用原子层 skill（如 `feishu-doc-fetch`），必须通过 Pipeline 流程。
3. 根据 `entry-router` 的路由结果触发对应 Pipeline（如 `dev-flow`、`hotfix-flow`）。

正确流程：

```
用户输入 ──► entry-router ──► Pipeline ──► 组合层 ──► 原子层
```

错误流程：

```
用户输入 ──► 直接调用 feishu-doc-fetch ❌
```

### 3.1 先计划，后执行

AI 在进入代码修改前，必须先输出计划或修复方案。

要求：

1. 先识别目标、范围、影响文件、风险点。
2. 输出任务拆分、执行顺序、验证方式。
3. 未完成计划前，不得直接改代码。

### 3.2 计划生成后默认暂停

计划输出完成后，AI 必须暂停，并等待用户明确确认。

允许进入实现阶段的标准口令：

- `确认计划，开始执行`
- `确认修复，开始执行`

未收到确认前：

- 不得修改代码
- 不得执行计划内任务
- 只能补充说明、答疑或更新计划

### 3.3 只按当前计划执行

进入实现阶段后，AI 仅允许执行当前计划中已列出的事项。

禁止行为：

- 顺手重构无关模块
- 擅自补充计划外需求
- 在未说明的情况下扩大影响范围
- 把优化建议直接实现为代码改动

### 3.4 偏差先更新计划

如果执行中发现以下情况，必须先暂停再继续：

- 实际代码结构与预期不一致
- 计划遗漏必要任务
- 需要新增依赖、脚本、配置或接口
- 修复范围扩大或影响到更多模块

处理顺序必须是：

1. 说明偏差原因
2. 说明影响范围
3. 更新计划或修复方案
4. 等待确认或在已授权前提下继续执行

### 3.5 完成一项，汇报一项

AI 在执行计划时，必须提高汇报频率。

每完成一个任务，都必须同步说明：

- 当前完成的是哪一项
- 修改了哪些文件
- 做了什么改动
- 验证结果是什么
- 下一步准备做什么

### 3.6 阶段结束必须汇总

每个阶段结束后，AI 必须输出阶段总结，至少包括：

- 当前阶段名称
- 已完成内容
- 当前进度
- 是否存在风险或阻塞
- 下一步动作

## 4. 标准执行节奏

### 阶段 A：需求获取 / 问题定位

AI 应先完成：

- 需求识别
- 输入类型判断
- 范围梳理
- 分支或问题信息提取

输出重点：

- 需求摘要
- 分支信息
- 下一步建议

### 阶段 B：计划模式

AI 应输出：

- 任务清单
- 执行顺序
- 执行边界
- 本次不做项
- 风险点
- 待确认事项

计划模式结束后，必须明确提示：

- 当前只输出计划，不改代码
- 需要用户回复 `确认计划，开始执行`

### 阶段 C：实现模式

AI 应严格按计划逐项推进。

每项任务都建议按以下节奏汇报：

1. 任务名称
2. 修改文件
3. 已完成改动
4. 验证结果
5. 下一步

### 阶段 D：验证 / 交付

AI 应汇报：

- 编译或测试结果
- 审查结果
- 提交/推送/部署结果
- 剩余人工操作项

## 5. 推荐汇报模板

### 5.1 计划生成后

```text
计划已生成，当前不执行代码修改。

执行边界：
- 本次包含：...
- 本次不包含：...

待确认事项：
- ...
- ...

请回复：确认计划，开始执行
```

### 5.2 执行中单项汇报

```text
[2/5] 已完成：实现 Service 层
- 修改文件：a.java, b.java
- 改动说明：补充查询逻辑与参数校验
- 验证结果：编译通过 / 单测通过
- 下一步：实现 Controller 层
```

### 5.3 发现偏差时

```text
发现执行偏差，当前已暂停。
- 偏差原因：...
- 影响范围：...
- 建议处理：先更新计划，再继续执行
```

### 5.4 阶段完成后

```text
当前阶段：代码实现
- 已完成：3/5 项任务
- 当前状态：按计划推进中
- 风险/阻塞：无
- 下一步：继续执行第 4 项任务
```

## 6. 热修复补充规则

热修复允许使用简化方案，但不能跳过确认与汇报。

额外要求：

- 先输出修复摘要：根因判断、影响范围、最小修复步骤
- 只修当前故障，不顺带优化
- 每完成一个修复动作立即汇报
- 若影响范围扩大，先更新修复方案

热修复确认口令：

- `确认修复，开始执行`

## 7. 用户推荐用法

如果希望 AI 更稳定地遵守本规则，建议直接使用如下指令：

```text
请严格按 AI DevCopilot 标准流程执行：
1. 先进入计划模式，只输出计划，不要直接改代码；
2. 等我回复“确认计划，开始执行”后再进入实现；
3. 执行时只按当前计划推进，不要跳步骤，不要做计划外修改；
4. 如果发现偏差，先更新计划再继续；
5. 每完成一项必须向我汇报：做了什么、改了哪些文件、验证结果和下一步。
```

## 8. 生效原则

当本规则与自由发挥式对话冲突时，优先遵循本规则。

如果用户明确要求缩短流程，也至少保留以下底线：

- 先说明计划或修复方案
- 关键阶段需要确认
- 不做计划外修改
- 完成一项汇报一项
