# Writing Plans (生成计划)

这个技能用于根据需求生成标准化的执行计划文档。

## 输入标准化

```yaml
title: 需求标题
issue_id: Issue ID
type: feature | fix | refactor
description: 需求描述
details: 详细需求内容
```

## 输出标准化

- **格式**: Markdown 计划文档
- **路径**: 由编辑器自动管理
- **要求**: 必须包含任务清单、执行顺序、影响范围、风险点、待确认事项

## 执行流程

### 步骤 1: 分析需求

```
分析需求内容，识别：
- 涉及的模块/文件
- 技术方案选择
- 依赖关系
- 风险点
- 边界与不做项
```

### 步骤 2: 生成任务清单

```
拆分为可执行的任务：
- 每个任务有明确的输入/输出
- 任务间依赖关系
- 按执行顺序排列
- 标注需要用户确认的点
```

### 步骤 3: 生成计划文档

```
═══════════════════════════════════════════
📝 计划生成
═══════════════════════════════════════════

✓ 需求标题: 用户登录
✓ Issue: #23181
✓ 任务数: 5

📄 计划文档:
   计划已生成（由编辑器自动管理）

📌 待确认:
   - 是否按当前技术方案实施
   - 是否接受任务拆分与执行顺序

📊 状态: 计划已生成，等待确认
⏸️  下一步: 等待用户回复“确认计划，开始执行”
```

### 步骤 4: 请求确认

```
计划生成后必须暂停，明确提示：
- 当前仅输出计划，不执行代码修改
- 请用户确认后再进入执行阶段
- 如用户提出变更，先更新计划文档
```

## 计划文档模板

```markdown
# {需求标题}

> Issue: #{issue_id}
> 类型: {type}
> 创建时间: {date}

## 需求概述

{需求描述}

## 技术方案

### 方案设计
{技术方案描述}

### 涉及模块
- 模块A: 描述
- 模块B: 描述

## 任务清单

### 阶段 1: 数据层
- [ ] 1.1 创建 Entity/POJO
- [ ] 1.2 创建 Mapper/Repository
- [ ] 1.3 数据库迁移脚本

### 阶段 2: 服务层
- [ ] 2.1 实现 Service
- [ ] 2.2 单元测试

### 阶段 3: 接口层
- [ ] 3.1 创建 Controller
- [ ] 3.2 接口文档

## 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| ... | ... | ... |

## 执行边界

- 本次仅包含: {scope_in}
- 本次不包含: {scope_out}

## 待确认事项

- [ ] 技术方案确认
- [ ] 任务顺序确认
- [ ] 影响范围确认

## 验收标准

- [ ] 功能验收
- [ ] 测试覆盖
- [ ] 代码审查
```

## 计划规则

1. **先界定边界**: 必须明确本次要做什么、明确不做什么。
2. **任务可汇报**: 任务拆分应细化到可单独汇报完成结果。
3. **等待确认**: 计划生成后默认暂停，不得直接进入代码修改。
4. **变更先改计划**: 用户调整方案或范围时，先更新计划文档。
