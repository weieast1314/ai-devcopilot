# 快速参考：用户视角命令速查

> 这份文档面向日常使用者，重点回答“我现在应该怎么对 AI 说”。

---

## 1. 推荐原则

- 优先使用自然语言，不必强记所有内部 Skill 名称
- 只有在你想显式触发某个流程时，再使用 `/xxx` 形式命令
- 先计划、后执行；计划确认前不建议直接要求改代码

---

## 2. 最常用输入

| 目标 | 推荐输入 | 直接触发词 / 兼容输入 | 说明 |
|------|----------|----------------------|------|
| 启动标准开发 | `开始开发` | `/dev` | 进入标准开发流程 |
| 启动热修复 | `热修复`、`线上问题需要紧急修复` | `/hotfix` | 进入热修复流程 |
| 从飞书开始 | `帮我实现这个需求：https://feishu.cn/wiki/xxx` | 粘贴飞书链接 | 自动读取需求并进入开发流 |
| 从文字需求开始 | `#23181 需要新增用户登录功能` | 自然语言需求描述 | 自动提取需求并进入开发流 |
| 生成计划 | `进入计划模式` | `生成计划`、`写计划` | 先产出方案，不直接改代码 |
| 开始执行 | `确认计划，开始执行` | `执行计划` | 按已确认计划逐项实现 |
| 热修复确认 | `确认修复，开始执行` | - | 按修复方案开始执行 |
| 代码验证 | `验证代码` | `代码验证`、`/code-verification` | 编译 / 测试 / 审查 |
| 代码交付 | `代码交付` | `/code-delivery` | 提交、推送、部署或 PR |
| 代码审查 | `代码审查` | `Review` | 检查规范与风险 |

---

## 3. 按场景速查

### 场景 A：拿到需求，想开始开发

推荐说法：

```text
开始开发
```

或：

```text
帮我实现这个需求：https://feishu.cn/wiki/xxx
```

或：

```text
#23181 需要新增用户登录功能
```

### 场景 B：想先出方案，不直接改代码

```text
进入计划模式
```

你应该期待 AI：
- 输出任务清单
- 说明执行边界
- 列出风险点和待确认事项
- 暂停等待你确认

### 场景 C：方案没问题，开始写代码

```text
确认计划，开始执行
```

或：

```text
执行计划
```

### 场景 D：我只想做验证

```text
验证代码
```

### 场景 E：我想提交并推送

```text
代码交付
```

或：

```text
/code-delivery
```

### 场景 F：线上故障需要快速处理

```text
热修复
```

或：

```text
线上问题需要紧急修复
```

修复方案出来后，继续说：

```text
确认修复，开始执行
```

## 4. 推荐说法 / 兼容说法 / 不推荐说法

| 类型 | 示例 | 说明 |
|------|------|------|
| 推荐说法 | `开始开发`、`进入计划模式`、`验证代码`、`代码交付` | 当前文档统一推荐 |
| 兼容说法 | `/dev`、`/hotfix`、`/code-delivery` | 适合喜欢显式命令的用户 |
| 不再推荐 | `/dev-flow`、`完成分支`、`/finish-branch` | 旧文档里可能看到，当前不建议继续传播 |

---

## 5. 一个最小可用流程

```text
用户: 帮我实现这个需求：https://feishu.cn/wiki/xxx
AI:   [读取需求并生成计划]

用户: 确认计划，开始执行
AI:   [按计划逐项修改并汇报]

用户: 验证代码
AI:   [输出验证结果]

用户: 代码交付
AI:   [提交、推送，并按选择执行交付]
```

---

## 6. 中断恢复模板

如果会话中断或你换了设备，可以直接复制下面的话继续：

```text
继续刚才的任务：当前分支 feat/23181-login，计划已确认，已经完成接口层，继续执行 Service、验证和交付。
```

如果你不知道当前停在哪一步，至少补充：
- 当前分支
- 计划是否已确认
- 已完成项 / 未完成项
